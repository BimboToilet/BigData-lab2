## Запуск
`DATASET_SIZE=<регулируемое количество строк в входном файле> REQUIRED_DN=<количество Hadoop DataNodes> OPTIMIZE=<булево значение, оптимизировать ли вычисления в Spark> docker compose --profile <1dn/3dn (два профиля с 1 и с 3 DataNodes соответственно, должен совпадать с REQUIRED_DN)> up -d`  
  
## Принцип работы
Сначала запускается NameNode (доступ к веб-панели Hadoop через 0.0.0.0:9870), параллельно с ним запускается Spark Master (доступ к веб-панели через 0.0.0.0:8080). После успешного запуска Spark Master запускается Spark Worker (используется один).  
После запуска NameNode запускается указанное количество DataNodes.  
  
Одновременно с ними запускается контейнер, генерирующий входные данные с указанным количеством строк. В данных 6 столбцов с разными типами данных: даты, булевы значения, целые и вещественные числа, категориальные и порядковые признаки.  
  
После успешной отработки генератора, а также запуска всех перечисленных контейнеров, запускается главный менеджер. Он проверяет доступность Spark и HDFS, загружает данные в HDFS, удаляя их локально. Затем запускается пайплайн обработки данных в Spark.  
  
## Ограничения
Для Hadoop выставлен размер блока в 64 МБ, ограничение в оперативной памяти для каждого DataNode - 512 МБ. Этого достаточно для генерируемых датасетов. Далее для каждого эксперимента будут рассмотрены результаты работы на данных из 1, 2.5 и 5 млн. строк.  
  
Для Spark Worker ограничение в оперативной памяти - 1 ГБ, в приложении указано, что каждый исполнитель запрашивает максимально 512 МБ.  
  
Так что на одном Spark Worker скорее всего запустится лишь один исполнитель.  
  
## Эксперименты
Было рассмотрено 4 конфигурации: 1 DataNode и Spark без оптимизаций, 3 DataNodes и Spark без оптимизаций, и то же самое, только с оптимизацией Spark. Под оптимизацией подразумевается использование `.cache()` и `.repartion()`.  
  
Рассматривалось как общее время выполнения, так и отдельно время на загрузку данных, время на применение оптимизации, время на сами вычисления отдельно. Также сравнивалось количество используемой исполнителями оперативной памяти.  
  
<img width="1068" height="542" alt="Общее время выполнения" src="https://github.com/user-attachments/assets/f2bbf08b-15c2-4d98-a1c1-afbcec0a3b12" />  
  
Это суммарный результат отдельных стадий работы.  
Конфигурации с оптимизацией на больших данных работают в 2.5–3 раза медленнее, чем без оптимизаций.  
Разница между 1 DN и 3 DN минимальна. Это объясняется тем, что здесь используется всего один исполнитель.  
Сколько бы копий данных ни лежало в HDFS, обрабатывает их один и тот же вычислительный узел, хотя положительное влияние все равно есть.  
  
<img width="1176" height="599" alt="Время вычислений" src="https://github.com/user-attachments/assets/75813bb9-076b-46c4-81c1-62c77144c879" />  
  
Без оптимизации время растет линейно и остается относительно низким, тут происходит чтение данных и сразу делаются вычисления.  
С оптимизацией время выполнения основного action значительно выше.  
Хотя данные в кэше, накладные расходы на предварительный shuffle (repartition) и управление памятью при переполнении кэша полностью нивелируют выгоду.  
На 5 млн. строк время экспоненциально растет, так как Spark тратит ресурсы на сбор мусора, пытаясь очистить память под новые задачи.  
  
<img width="1291" height="657" alt="Время на оптимизацию" src="https://github.com/user-attachments/assets/ede75d24-fc64-4880-9e86-595969880126" />  
  
Здесь видно, что без оптимизации никакое время не тратится (этот блок пропускается).  
С оптимизацией же выполняется repartition - это shuffle, дорогая операция.  
  
<img width="1290" height="657" alt="Потребление RAM" src="https://github.com/user-attachments/assets/3f5c5d48-65c6-4c84-9ccf-8efebd7e56f5" />  
  
`cache()` и `count()` заставляют Spark немедленно вычислить все данные и попытаться перенести их в оперативную память.  
На графике RAM виден резкий скачок для оптимизированных версий. На 5 млн. строк кривая RAM падает. Это сигнализирует о том, что данные перестали помещаться в отведенную память (было выделено всего 512 МБ на исполнитель) и начали вытесняться на диск, либо Spark перестал кэшировать новые блоки.  
Последние два графика явно говорят, что "оптимизация" в данном контексте плохой вариант. Для такого объема данных и малого количества ресурсов кэширование приносит больше вреда, чем пользы.  
  
<img width="1291" height="649" alt="Время на загрузку данных из HDFS" src="https://github.com/user-attachments/assets/97a92500-8291-4c95-a329-4f06e53583dc" />  
  
График выглядит хаотично, для всех экспериментов время примерно одинаковое.  
В нашем случае при считывании происходит только реальное чтение, которое довольно быстрое. Подготовки схемы не происходит, так как она заготавливается нами заранее.  
Небольшие колебания на графике вызваны сетевыми задержки Docker-сети, временем на установку соединения с NameNode и/или другими внешними причинами (фоновыми вычислениями). На таких объемах HDFS не успевает показать свои преимущества.  
  
В данном случае "оптимизация" убивает производительность. repartition и cache имеют смысл, если используется один и тот же DataFrame в десятках разных вычислений. На разовой агрегации он не успевает показать свои преимущества.  
Чтобы увидеть ощутимую пользу от 3 DataNodes, нужно увеличить количество исполнителей и/или увеличить им ресурсы, чтобы Spark мог читать данные параллельно из разных блоков.

По данным графикам видно, что в целом везде время немного прибавилось. Видно ощутимое негативное изменение в времени на загрузку из HDFS. Это связано с дополнительными действиями по взаимодействию между исполнителями.  
Другими словами, параллелизация на таком объеме тоже не дает ощутимых преимуществ. Самым лучшим вариантом в данном случае является самый простой - "1 DataNode без оптимизации" с одним Spark Worker. Он прекрасно подходит для обработки такого объема данных. Это объясняется отсутствием лишних накладных расходов на общение и координацию между несколькими узлами.  
Однако, как уже было выяснено ранее, ему можно добавить дополнительный объем оперативной памяти для лучшей производительности.
